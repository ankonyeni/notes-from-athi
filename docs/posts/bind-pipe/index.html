Introduction

R is remarkably conservative regarding core syntax. While the surface suggests a vast array of syntactic forms (arithmetic operators, indexing brackets, and control flow), R’s elegance lies in its exceptionally small grammar.

Most Syntax Is Function Calls

The defining characteristic of this grammar is that almost everything is a function call in disguise. In R, 1 + 2 is functionally equivalent to the backticked call `+`(1, 2). Even constructs like if or for are internal functions that return values. This ability to inspect and manipulate code as language objects allows R to remain flexible without a sprawling list of hard-coded rules.

Some Syntax Is Parser-Only

However, the famous adage that “in R, everything is an object” is only partially true. While most of R’s power stems from objects and function calls, a handful of symbols exist as pure grammar. These are parsing instructions rather than functions. For instance, the right-assignment operator -> cannot be called as a function; the parser immediately rewrites 1 -> x as x <- 1 before execution. Because these constructs exist only as rules for the parser, they lack the object status that defines the rest of the language.

quote(1 -> x)
#> x <- 1

The Native Pipe: R Joins the Show

With R 4.1.0, the language officially introduced the native pipe operator |>. While R was late to this style compared to functional peers like F#, Elixir, and OCaml, its implementation was heavily influenced by the massive success of the magrittr package. For years, the %>% operator was the de facto standard, proving a deep appetite for linear, readable data transformations over deeply nested expressions:

# Nested
head(transform(subset(mtcars, cyl == 6), kmpl = mpg * 0.425144))

# Linear
mtcars |>
  subset(cyl == 6) |>
  transform(kmpl = mpg * 0.425144) |>
  head()

How the Native Pipe Works

Unlike magrittr, which operates as a function at runtime, the native pipe is a compile-time transformation. When the R parser encounters x |> f(y), it does not run a pipe function. Instead, it physically rewrites the expression into f(x, y) before any code is executed. This lack of overhead makes the native pipe more efficient and ensures stack traces remain clean.

quote(iris |> head())
#> head(iris)

The Grammar of the Placeholder _

Because the native pipe is pure grammar, it requires a syntactic way to handle cases where data should not go into the first argument. R 4.2.0 introduced the placeholder _ for this purpose.

Like the pipe itself, the underscore is not an object; you cannot assign a value to _ or use it in a standard calculation. It is a coordinate for the parser. When the parser sees _, it recognizes it as a signal to move the left-hand side into that specific named argument.

quote("Hello World" |> grepl('W', x = _))
#> grepl("W", x = "Hello World")

Syntax vs. Functions: Why This Distinction Matters

It is critical to distinguish between these two implementations. The magrittr pipe %>% and its dot placeholder . are runtime constructs that rely on complex environment management. Under the hood, magrittr’s C code treats the dot as a SYMSXP (a symbol object). As each step of the pipe evaluates, the code manually binds and updates this symbol in a dedicated execution environment using Rf_defineVar. A variable named . is created and repeatedly overwritten to pass data forward.

In contrast, the native pipe |> and the _ placeholder are not objects or variables at all; they are parsing instructions. The underscore is a special token that exists only as a coordinate for the parser. When you write x |> f(y = _), the parser physically rewrites the expression into f(y = x) before the code is even compiled or executed. The pipe and the placeholder effectively disappear during the transition from source code to the Abstract Syntax Tree (AST), leaving the engine to execute standard, optimized function calls with no internal “variable shuffling.”

While the native pipe |> is a powerful tool for streamlining code, its design as a strict syntactic rewriter introduces structural hurdles. Because it is a simple instruction for the parser rather than a sophisticated function, it lacks the flexibility required for certain complex data transformations.

The Limitations of the Native Pipe

The primary limitations of |> stem from its rigidity as pure grammar:

  • The “Single Use” Placeholder Constraint: In the native pipe, the underscore placeholder _ can only be used once per call. If you need to reference the piped data multiple times—for example, to compare two columns or use the data in multiple arguments—the parser throws an error.
  • The “Named Argument” Requirement: The placeholder _ must be passed to a specifically named argument (e.g., data = _). You cannot use it as a positional argument, inside a nested expression, or within a mathematical operation without wrapping the entire step in a verbose anonymous function.
  • The “Function Call” Mandate: The native pipe strictly expects the right-hand side to be a function call. It cannot pipe into a symbol or a raw expression. For example, 1:10 |> {_ * 1} is syntactically invalid because the right-hand side is not a formal call.

Discovering the Hidden Grammar

These limitations often force users back toward the magrittr package. However, while browsing through the R source code, I noticed an experimental feature hidden within the language’s own grammar.

Spotting => in R Source

I was looking into the core grammar file src/main/gram.c in the R source, where the fundamental rules of how R interprets text are defined. Amidst the low-level logic that handles assignment and control flow, the symbol => caught my eye. It wasn’t a standard operator I had seen in any documentation, but there it was, baked into the very foundation of the R parser as an experimental feature.

Tracing the First Discussion

Curiosity piqued, I did some digging to find out when this was first proposed. According to my investigation, I traced it back to January 2021. Based on the R-devel mailing list archives, this is one of the earliest discussions I could find regarding the operator, where it was debated as a way to provide a more flexible “binding” mechanism.

The Experimental Pipe-Bind =>

The experimental pipe bind (=>) is designed to clear the structural hurdles inherent in the native pipe. If the standard pipe is about redirection, the pipe bind is about lexical binding.

Instead of a “blind” injection into a single argument slot, the pipe bind allows you to bind the left-hand side to a temporary name (a variable) that exists only for the scope of the next expression.

How the Pipe Bind Works

Like the pipe operator, the pipe bind is implemented purely as grammar. When the parser encounters the pipe bind syntax x |> y => expr, it follows a specific transformation rule:

  1. It takes the object on the left (x).
  2. It creates a temporary, anonymous function where the name you provided (y) is the argument.
  3. It places the expression on the right (expr) as the body of that function.
  4. It immediately calls that function using x as the input.

Essentially, x |> y => expr is a syntactic shortcut for:

(function(y) expr)(x)
Important

To try out this feature in R, you must first set the following environment variable:

Sys.setenv("_R_USE_PIPEBIND_" = "TRUE")

Otherwise, you will encounter the following error when attempting to use the operator:

Error: '=>' is disabled; set '_R_USE_PIPEBIND_' envvar to a true value to enable it (<input>:1:11)

How it Overcomes the Pipe’s Hurdles

The pipe bind transforms the logic from structural rewriting to a temporary local scope:

Multiple Uses

Since the data is bound to a name, you can reference it as many times as you want.

data <- c(12, 9, 11, 13, 10)
data |> x => data.frame(
    value = x, 
    diff = x - mean(x)
)
#>   value diff
#> 1    12    1
#> 2     9   -2
#> 3    11    0
#> 4    13    2
#> 5    10   -1

Positional Freedom

Because the right-hand side is treated as a scope where the data is simply a variable, you can use the bound data in any positional argument without the need to specify a parameter name (like x = _).

# No need for named arguments; the variable can be passed positionally
"hello" |> string => gsub("h", "j", string)
#> [1] "jello"

Expression Freedom

The pipe bind does not require a formal function call on the right-hand side. You can pipe directly into raw expressions or braced blocks, bypassing the need for a strict function structure.

data <- c(12, 9, 11, 13, 10, 14, 8, 12)

data |> x => {
    x_sum <- sum(x)
    n <- length(x)
    x_sum/n
}
#> [1] 11.125

Because the pipe bind binds the incoming lhs to a specific variable name, the rhs is no longer restricted to being a function call. It can be any valid R expression.

mtcars[mtcars$cyl == 9, ] |> df => {
    if (nrow(df) > 0) {
        lm(mpg ~ wt, data = df)
    } else {
        message("No records found to run the model.")
    }
}
#> No records found to run the model.

Conclusion: A Glimpse into R’s Future

The => operator is an interesting addition to R’s syntax. While the native pipe |> streamlines linear code through simple rewriting, the pipe bind addresses the need for more flexible plumbing: referencing the same dataset multiple times and performing inline operations.

However, it is important to remember that this is an experimental feature. It remains hidden behind an environment variable because it is still a “draft” within the language’s source code. There is no guarantee that it will be promoted to a stable, production-ready release. If it does eventually graduate, both its syntax and its underlying behavior may change to ensure long-term compatibility with the rest of the language.

For now, the pipe bind offers a look at how R might further bridge the gap between simple piping and complex functional programming. Whether it remains hidden in the source code or becomes a standard feature, it demonstrates a thoughtful approach to making R’s grammar more expressive for modern data workflows.
